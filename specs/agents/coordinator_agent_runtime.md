# 协调智能体说明

## 1. 角色定位

在本文第 2 章对应的系统总体架构中，协调智能体应当被理解为运行时框架中的唯一流程控制中心，而不是某个具体建模阶段的执行者。  
它的任务不是直接提取特征或生成几何，而是接收统一目标对象，决定下一步该交给哪个执行智能体，并在执行结果返回后判断流程是否继续、是否进入评估与优化，或者是否可以结束本轮运行。

因此，协调智能体在系统中的定位，是“中心化调度者”。

## 2. 所属链路

协调智能体所在的主链路如下：

1. 用户通过 `UserConsole` 提交请求  
2. 请求被整理成统一的 `UserGoal` 对象  
3. `AgentRuntime` 创建初始任务并交给协调智能体  
4. 协调智能体按当前状态把任务分发给感知、建模、评估或优化智能体  
5. 执行结果返回协调智能体，由它决定下一步  
6. 满足停止条件后，由协调智能体结束本轮运行

这意味着系统不是执行智能体之间点对点自由通信，而是始终通过协调智能体完成中心化衔接。

## 3. 主要职责

协调智能体在这条运行时主链里主要承担四类职责。

第一，是接收统一目标。  
它接收已经格式化好的 `UserGoal`，并把用户想要的结果转换成系统内部可执行的推进起点。

第二，是分发内部任务。  
它根据当前任务类型、共享状态和上一轮结果，决定下一步交给 `PerceptionAgent`、`ModelingAgent`、`EvaluationAgent` 还是 `OptimizationAgent`。

第三，是汇总阶段结果。  
每个执行智能体完成后，协调智能体负责接收 `AgentResult`，把关键产物、评分和决策写回共享状态。

第四，是控制流程闭环。  
它决定系统是继续迭代、进入下一执行环节，还是结束运行，因此它是唯一有权收束整个运行过程的角色。

## 4. 输入内容

协调智能体在系统主链中主要接收以下输入：

- 用户目标对象 `UserGoal`
- 统一任务对象 `AgentTask`
- 执行结果对象 `AgentResult`
- 运行时共享状态 `RuntimeStateSnapshot`
- 评估分数、优化建议和产物路径

这些输入说明，协调智能体依赖的是统一协议和运行时状态，而不是单独某一阶段的提示词资产。

## 5. 输出内容

协调智能体在系统主链中的主要输出包括：

- 下一步内部任务
- 当前流程决策
- 当前轮次的状态更新
- 运行结束判断
- 最终结果汇总

换句话说，它输出的是“系统接下来怎样推进”，而不是某个局部几何结果本身。

## 6. 上下游关系

上游：

- 用户控制入口 `UserConsole`
- 运行时框架 `AgentRuntime`
- 统一目标与状态对象

下游：

- 感知智能体
- 建模智能体
- 评估智能体
- 优化智能体
- 结果记录层

因此，协调智能体位于用户目标与执行角色之间，负责把系统组织成可持续运行的中心化结构。

## 7. 工程对应文件

当前与协调智能体最相关的工程文件包括：

- `console/user_console.py`
- `runtime/agent_runtime.py`
- `agents/coordinator_agent.py`
- `models/agent_protocol.py`
- `runtime/message_bus.py`
- `specs/agent_system_status.md`

这些文件共同构成了第 2 章所描述的“统一目标、统一协议、统一调度”的工程载体。

## 8. 边界说明

协调智能体不直接负责：

- 提取 STL 结构特征
- 生成 STEP 或 STL 模型
- 执行几何评估计算
- 输出优化参数本体

它负责的是：

- 决定下一步由谁执行
- 判断结果是否需要继续修正
- 维护运行闭环
- 统一收束整轮执行过程

因此，协调智能体的核心价值不在于直接完成建模动作，而在于维持整套系统的中心化控制和状态一致性。
